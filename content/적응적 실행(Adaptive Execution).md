[[지능형 AI 위임(Intelligent AI Delegation)|지능형 AI 위임]] 프레임워크의 다섯 가지 기둥 중 하나인 **적응적 실행(Adaptive Execution)**은, 위임 결정이 정적이어서는 안 되며 환경 변화, 자원 제약, 하위 시스템의 장애에 적응해야 한다는 원칙입니다.

---

## 핵심 개념

기존의 AI 에이전트 위임 시스템은 "한 번 맡기면 결과만 기다리는" 정적 방식이었습니다. 적응적 실행은 이와 달리, **실행 도중에도 상황을 감시하고 문제가 생기면 에이전트를 교체하거나 계획을 수정하는 유연한 실행 방식**을 의미합니다.

### 비유: 택배 배송

택배 배송에 비유하면 차이가 명확해집니다.

**정적 실행 (기존 방식):**
- "A 배송기사에게 맡겼으니 끝날 때까지 기다린다"
- A가 사고가 나든, 길이 막히든, 그냥 기다림

**적응적 실행 (이 논문의 제안):**
- A에게 맡겼지만 **실시간으로 상태를 감시**
- A가 교통체증에 막혀 지연되면 → 근처의 B 기사에게 **도중에 교체**
- 갑자기 폭우가 오면 → 배송 경로를 **재조정**
- A의 차량이 고장나면 → 즉시 C에게 **재위임**

인간 조직의 프로젝트 관리에서는 이미 자연스러운 방식입니다. 이 논문은 AI 에이전트 간 위임에서도 동일한 수준의 **지속적 감시와 실시간 대응**이 필요하다고 주장합니다.

---

## 무엇에 적응하는가?

적응적 실행이 반응해야 하는 트리거는 두 가지로 나뉩니다.

### 외부 트리거 — "세상이 바뀌었다"

위임자나 수임자가 통제할 수 없는 환경의 변화입니다.

- **작업 명세 변경 또는 취소**: 위임자가 요구사항을 수정하거나 작업 자체를 철회
- **외부 자원 가용성/비용 변동**: API 장애, 데이터셋 접근 불가, 컴퓨팅 비용 급등
- **더 높은 우선순위의 새 작업**: 긴급 작업이 들어와 기존 작업을 선점
- **보안 위협 식별**: 보안 시스템이 수임자의 악의적 행위를 감지

### 내부 트리거 — "일하는 쪽에 문제가 생겼다"

수임자의 실행 과정에서 발생하는 문제입니다.

- **성능 저하**: 약속한 SLA(서비스 수준 합의)를 미달
- **예산 초과**: 할당된 자원 한도를 넘어섬
- **중간 산출물 검증 실패**: 작업 중간 결과가 품질 기준을 통과하지 못함
- **수임자 응답 불능**: 에이전트가 응답하지 않거나 작동을 멈춤

## 트리거 감지 후 어떻게 대응하는가?

문제가 감지되면 **4단계 적응적 대응 주기**가 시작됩니다.

```
1단계. 감시(Monitor)   → 지속적 모니터링으로 문제를 식별한다
2단계. 진단(Diagnose)  → 근본 원인을 분석하고 대응 시나리오를 평가한다
3단계. 대응(Respond)   → 긴급도에 따라 단계적으로 조치한다
4단계. 상향 보고(Escalate) → 필요시 인간에게 알린다
```

### 3단계 대응의 세 가지 수위

긴급도에 따라 대응 강도가 달라집니다.

| 수위 | 대응 | 예시 |
|------|------|------|
| 경미 | **매개변수 조정** | 시간 제한을 늘려주거나, 메모리 할당을 증가 |
| 중간 | **재위임** | 현재 수임자를 다른 에이전트로 교체 |
| 심각 | **완전 재분해** | 작업 자체를 다시 쪼개어 새로운 실행 계획 수립 |

## 핵심 원칙: 가역성이 대응 수위를 결정한다

적응적 실행에서 가장 중요한 판단 기준은 **작업의 가역성(reversibility)**입니다.

| 작업 유형 | 예시 | 실패 시 대응 |
|-----------|------|-------------|
| **가역적 작업** | 코드 생성, 데이터 변환, 문서 작성 | 자동으로 다른 에이전트에게 재위임. 실패해도 되돌릴 수 있으므로 부담이 적음 |
| **비가역적 작업** | 결제 처리, 데이터 삭제, 외부 API 호출 | 즉각 중단하거나 인간에게 상향 보고. 자동 재시도하면 돌이킬 수 없는 피해가 발생할 수 있음 |

코드를 잘못 생성했다면 다시 생성하면 됩니다. 하지만 결제를 잘못 처리했다면 되돌리기가 훨씬 어렵습니다. 이 차이가 시스템이 얼마나 공격적으로 자동 대응할 수 있는지를 결정합니다.

## 조율 방식: 중앙집중형 vs 분산형

적응적 실행을 누가, 어떻게 조율하느냐에 따라 두 가지 접근법이 있습니다.

| 방식 | 장점 | 단점 |
|------|------|------|
| **중앙집중형** | 전담 위임자가 전역적 관점을 유지하여 일관성 있는 조율 가능 | 단일 장애점(SPOF), 계산적 병목 |
| **분산형 (시장 기반)** | 재위임 요청을 경매에 올려 유연하게 대체 수임자 확보 | 조율 비용, 시장 불안정성 |

분산형 시장 기반에서는 불이행 에이전트에게 가격 차이를 벌칙으로 부과하고, Smart contract에 적응적 조정을 위한 사전 합의 조항을 포함할 수 있습니다.

## 안정성 조치

적응적 실행이 너무 빈번하게 발동되면 오히려 시스템을 불안정하게 만들 수 있습니다. 논문은 이를 방지하기 위한 세 가지 안정성 조치를 제안합니다.

- **냉각 기간(cooldown period)**: 재입찰까지 최소 대기 시간을 두어 과도한 교체를 방지
- **감쇠 계수(damping factor)**: 평판 업데이트를 점진적으로 반영하여 일시적 실패로 인한 급격한 평판 하락을 완화
- **수수료 인상**: 빈번한 재위임에 비용을 부과하여 불필요한 교체를 억제

이는 인간 조직에서도 마찬가지입니다. 프로젝트 담당자를 너무 자주 바꾸면 인수인계 비용이 누적되어 오히려 전체 효율이 떨어지는 것과 같은 원리입니다.
