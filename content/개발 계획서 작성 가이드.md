---
title: "개발 계획서, 언제 어떻게 작성해야 할까?"
date: 2025-07-03
---

많은 개발자들이 "일단 코딩부터 시작하자"는 유혹에 빠지곤 합니다. 특히 빠르게 변화하는 스타트업이나 애자일 환경에서는 문서 작업이 불필요한 시간 낭비처럼 느껴질 수 있습니다. 하지만 잘 작성된 개발 계획서는 단순한 서류 작업을 넘어, 프로젝트를 성공으로 이끄는 나침반 역할을 합니다.

이 글에서는 개발 계획서가 왜 필요한지, 언제 작성하고 언제 생략하는 것이 좋은지, 그리고 모두에게 도움이 되는 효과적인 계획서는 어떻게 작성하는지에 대한 가이드라인을 제시합니다.

## 개발 계획서의 필요성

개발 계획서는 단순히 '무엇을 만들 것인가'를 넘어, 프로젝트의 성공 가능성을 높이는 여러 순기능을 가지고 있습니다.

### 1. 목표의 명확화와 팀 정렬
계획을 수립하는 과정에서 프로젝트의 목표, 범위, 그리고 제약 조건이 명확해집니다. 개발자, 기획자, 디자이너 등 모든 팀원이 동일한 목표를 바라보게 되어 불필요한 오해와 재작업을 줄일 수 있습니다.

### 2. 리스크 식별 및 관리
"이 기술은 우리 팀에 생소한데?", "외부 API 연동에 문제가 생기면 어떡하지?" 와 같은 잠재적 리스크를 미리 식별하고 대응 방안을 고민하게 합니다. 이는 프로젝트가 예상치 못한 암초에 부딪히거나, 관리되지 않는 [[기술 부채(Technical Debt)]]가 쌓이는 것을 막아줍니다.

### 3. 효율적인 자원 관리
프로젝트에 필요한 시간, 인력, 기술 스택 등을 예측하고 계획함으로써 자원을 효율적으로 배분할 수 있습니다. 이는 예산과 일정 준수의 기반이 되며, 전체적인 [[아키텍처 설계(Architecture Design)]]의 큰 그림 안에서 움직이게 도와줍니다.

### 4. 진행 상황 추적 및 예측 가능성 확보
잘 정의된 마일스톤과 작업 목록은 프로젝트가 계획대로 진행되고 있는지 측정하는 기준이 됩니다. 이를 통해 이해관계자들은 프로젝트의 현재 상태를 명확히 파악하고 미래를 예측할 수 있습니다.

### 5. 의사소통의 중심
개발 계획서는 프로젝트에 대한 모든 정보가 담긴 단일 진실 공급원(Single Source of Truth) 역할을 합니다. 새로운 팀원이 합류하거나 의사결정이 필요할 때 중요한 의사소통의 도구가 됩니다.

## 작성해야 할 때 vs. 신중해야 할 때

개발 계획이 항상 무겁고 상세한 문서일 필요는 없습니다. 상황에 따라 유연하게 접근해야 합니다.

### 계획서 작성이 꼭 필요한 경우
- **대규모 또는 복잡한 프로젝트**: 여러 팀과 기술이 얽혀 있어 명확한 청사진 없이는 혼란이 발생하기 쉽습니다.
- **장기 프로젝트**: 몇 개월 이상 소요되는 프로젝트는 명확한 로드맵이 없으면 방향을 잃기 쉽습니다.
- **높은 리스크를 가진 프로젝트**: 기술적 불확실성이 크거나 비즈니스 영향이 막대한 경우, 신중한 계획이 필수입니다.
- **외부 이해관계자와의 협업**: 고객사나 투자자에게 프로젝트의 신뢰성과 전문성을 보여주어야 할 때 중요합니다.

### 가벼운 계획 또는 생략을 고려할 경우
- **작은 규모의 작업 또는 버그 수정**: 계획 수립에 드는 비용이 실제 개발 비용보다 클 수 있습니다.
- **프로토타입 또는 PoC(Proof of Concept) 개발**: 목표가 빠른 실험과 학습일 때는 형식적인 계획보다 신속한 실행이 중요합니다. 계획이 오히려 창의성을 저해할 수 있습니다.
- **[[애자일 개발 방법론|애자일]] 성숙도가 높은 팀**: 애자일 팀은 장기간의 상세 계획 대신, 스프린트 계획, 백로그, 사용자 스토리 등 짧은 주기의 계획을 지속적으로 수행합니다. 이 경우 전통적인 개발 계획서는 애자일 철학에 맞지 않을 수 있습니다.

## 효과적인 개발 계획서 작성 가이드라인

효과적인 개발 계획서는 분량보다 내용의 질이 중요합니다. "쇼핑몰 리뷰 기능 개발" 프로젝트를 예시로 각 항목을 살펴보겠습니다. 전체 계획서의 완성된 모습이 궁금하다면, [[모범적인 개발 계획서 예시]] 문서를 참고해 주세요.

### 1. 목표(Goal)와 배경(Context) 정의
- **이 프로젝트를 왜 하는가?**: 해결하려는 문제와 기대 효과를 명확히 기술합니다.
    - **예시**:
        - **문제**: 현재 우리 쇼핑몰은 구매 전환율이 정체되어 있고, 제품에 대한 고객의 신뢰도가 낮다.
        - **기대 효과**: 사용자가 작성한 리뷰 기능을 도입하여 제품의 신뢰도를 높이고, 이를 통해 구매 전환율을 5% 향상시킨다.
- **배경**: 프로젝트가 나오게 된 비즈니스적, 기술적 배경을 간략히 설명합니다. 이때 팀 내에서 합의된 [[유비쿼터스 언어(Ubiquitous Language)]]를 사용하면 모두가 동일한 의미로 이해할 수 있습니다. (관련 문서: [[요구사항 명세서(Software Requirements Specification, SRS)]])
    - **예시**: 3분기 마케팅팀의 '고객 참여 증진 캠페인'과 연계하여 진행되며, 기존 데이터베이스 스키마에 리뷰 테이블을 추가하는 방식으로 개발한다.

### 2. 범위(Scope) 설정
- **포함될 기능(In-Scope)**: 이번 프로젝트에서 개발할 핵심 기능 목록을 구체적으로 정의합니다.
    - **예시**:
        - 사용자는 구매한 상품에 대해 텍스트와 별점(1~5점)으로 리뷰를 작성할 수 있다.
        - 작성된 리뷰는 상품 상세 페이지에 노출된다.
        - 사용자는 다른 사람의 리뷰에 '도움돼요' 버튼을 누를 수 있다.
- **제외될 기능(Out-of-Scope)**: 혼동을 피하기 위해 '하지 않을 일'을 명시하는 것이 매우 중요합니다.
    - **예시**:
        - 리뷰 작성 시 사진이나 동영상 첨부 기능은 이번 버전에 포함하지 않는다.
        - 리뷰에 대한 댓글 기능은 고려하지 않는다.
        - 비회원 리뷰 작성 기능은 지원하지 않는다.

### 3. 기존 시스템 제약사항 분석
- 새로운 기능을 추가하기 전에, 현재 시스템이 가진 기술적 제약 조건을 명확히 파악해야 합니다. 이는 나중에 발생할 수 있는 통합 문제를 예방하고, 더 현실적인 계획을 세우는 데 도움이 됩니다.
    - **DB 스키마**: 기존 테이블 구조와 관계를 변경해야 하는가? 변경이 어렵다면 어떻게 우회할 것인가?
    - **API 버전 및 통신 방식**: 기존 시스템은 RESTful API를 사용하는가, 아니면 gRPC를 사용하는가? 새로운 API는 기존의 통신 방식과 인증/인가 메커니즘을 따라야 한다.
    - **프론트엔드 제약**: 현재 프론트엔드 프레임워크(React, Vue 등)의 버전과 상태 관리 방식은 무엇인가? 새로운 UI 컴포넌트가 기존 시스템과 잘 통합될 수 있는가?
    - **인프라 환경**: 개발, 스테이징, 프로덕션 환경의 구성은 어떻게 되어 있는가? 배포 파이프라인은 어떻게 작동하는가?

### 4. 설계 및 작업 분할 (Design & WBS)
- 프로젝트를 관리 가능한 단위로 나누는 동시에, 설계 수준을 고려합니다. 이 과정은 보통 **고수준 설계(HLD)**와 **저수준 설계(LLD)**로 이어집니다.
- **[[아키텍처 설계(Architecture Design)|고수준 설계(HLD)]]**: 시스템의 전체 구조와 뼈대를 잡는 단계입니다. 개발 계획 단계에서는 어떤 아키텍처 패턴을 사용할지, 주요 기술 스택은 무엇인지 등을 결정합니다.
    - **예시**: 리뷰 기능을 위해 기존 모놀리식 아키텍처에 새로운 모듈을 추가하는 방식을 채택하고, 데이터베이스는 기존의 MySQL을 그대로 사용하기로 결정합니다.
- **[[상세 설계(저수준 설계)|저수준 설계(LLD)]]**: HLD에서 정의된 각 컴포넌트의 내부 동작을 구체화하는 단계입니다. 개발 계획서에는 LLD를 위한 구체적인 Task를 포함시킵니다.
    - **예시**:
        - **Epic: 리뷰 기능 개발**
            - **Story 1: 리뷰 기능 설계**
                - Task 1.1: 리뷰 기능 HLD (아키텍처 다이어그램 작성)
                - Task 1.2: 리뷰 기능 LLD (시퀀스, 클래스 다이어그램 작성)
            - **Story 2: 리뷰 기능 개발**
                - Task 2.1: 리뷰 테이블 DB 스키마 설계
                - Task 2.2: 리뷰 작성 API 개발
                - Task 2.3: 리뷰 작성 UI 페이지 개발
            - **Story 3: 리뷰 노출 기능**
                - Task 3.1: 상품별 리뷰 목록 조회 API 개발
                - Task 3.2: 상품 상세 페이지에 리뷰 UI 연동

### 5. 일정 및 마일스톤 수립
- 각 작업의 예상 소요 시간을 보수적으로 산정합니다. (예: 3일이 아닌 3~5일)
- 프로젝트의 주요 분기점이 되는 마일스톤(예: '로그인 기능 개발 완료', '알파 버전 출시')을 설정하여 중간 목표를 제시합니다.
    - **예시**:
        - **1주차**: 설계 완료 (HLD, LLD) 및 DB 스키마 확정 (마일스톤 1)
        - **2주차**: 백엔드 API 개발 완료
        - **3주차**: 프론트엔드 UI 개발 및 기능 통합 완료 (마일스톤 2)
        - **4주차**: QA, 버그 수정 및 최종 배포

### 6. 의존성 및 리스크 분석
- **기술 의존성**: 특정 기술, 라이브러리, 외부 API 등 프로젝트가 의존하는 요소를 명시합니다.
    - **예시**: 리뷰 작성 시 욕설 필터링을 위해 외부 '클린봇 API'를 사용하며, 해당 API의 응답 속도에 서비스 품질이 영향을 받을 수 있다.
- **팀/작업 의존성**: 'A 작업이 끝나야 B 작업을 시작할 수 있다'와 같은 작업 간의 의존 관계를 파악합니다.
    - **예시**: '리뷰 기능 LLD'가 완료되어야 백엔드와 프론트엔드 개발을 병렬로 진행할 수 있다.
- **리스크**: 예상되는 기술적, 비기술적 리스크와 그에 대한 완화 전략을 간략하게나마 준비합니다. 예를 들어, [[단위 테스트(Unit Test)]]나 [[통합 테스트(Integration Test)]]를 통해 코드의 안정성을 확보하는 것도 중요한 리스크 관리 전략입니다.
    - **예시**:
        - **리스크**: '클린봇 API' 장애 발생 시 욕설이 필터링되지 않음.
        - **완화 전략**: API 장애 시, [[서킷 브레이커(Circuit Breaker) 패턴]]을 적용하여 서비스 장애 전파를 막거나, 임시로 자체 키워드 기반 필터링 로직을 수행하는 방안을 마련한다.

### 7. 역할과 책임(R&R) 정의
- 누가 어떤 부분을 책임지고 있는지 명확히 하여 책임 소재를 분명히 하고 원활한 소통을 돕습니다.
    - **예시**:
        - **백엔드 개발 (API, DB)**: 김개발 팀장
        - **프론트엔드 개발 (UI)**: 이코딩 연구원
        - **QA**: 박테스트 주임
        - **프로젝트 총괄**: 최기획 PM

### 8. 살아있는 문서로 만들기
- 개발 계획서는 한 번 쓰고 버리는 문서가 아닙니다.
- 프로젝트 진행 상황에 따라 변경 사항을 꾸준히 업데이트하고, 팀 전체가 항상 최신 버전을 공유해야 합니다.
    - **예시**: '클린봇 API' 연동 테스트 중 심각한 성능 저하가 발견되어, 계획을 수정하여 자체 필터링 로직을 1순위로 개발하기로 결정. 해당 내용을 계획서에 즉시 반영하고 팀에 공유한다.

## 결론

개발 계획서는 프로젝트를 옥죄는 관료주의의 산물이 아니라, 불확실성의 안개 속에서 길을 밝혀주는 등대와 같습니다. 계획을 세우는 과정 자체만으로도 우리는 프로젝트를 더 깊이 이해하고 잠재적인 문제를 미리 발견할 수 있습니다. 잘 수립된 계획은 요구사항을 [[아키텍처 설계(Architecture Design)|고수준 설계]]와 [[상세 설계(저수준 설계)|저수준 설계]]로 구체화하고, 최종적으로는 [[코드의 유지보수성]]과 [[소프트웨어 설계의 유연성(Flexibility)]]을 높이는 첫걸음이 됩니다.

모든 프로젝트에 거창한 계획서가 필요한 것은 아닙니다. 프로젝트의 규모, 복잡성, 팀의 특성에 맞게 계획의 깊이와 형식을 조절하는 유연함이 중요합니다. 중요한 것은 '문서'가 아니라 '계획을 통한 공동의 이해와 정렬'이라는 본질을 잊지 않는 것입니다. 주기적인 [[회고(Retrospective)]]를 통해 계획을 점검하고 개선해 나가는 과정 역시 성공적인 프로젝트의 핵심 요소입니다. 최근에는 이러한 계획서 초안 작성에 도움을 주는 [[LLM을 활용한 효과적인 개발 계획서 작성법]]도 좋은 참고 자료가 될 수 있습니다.
