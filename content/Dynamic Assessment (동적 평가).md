동적 평가(Dynamic Assessment)는 [[AI 에이전트(AI Agent)|AI 에이전트]] 간 작업 위임에서, 수임자의 역량·신뢰성·상태를 **일회성 판단이 아닌 연속적 프로세스**로 추론하는 메커니즘을 의미합니다. Google DeepMind의 [[지능형 AI 위임(Intelligent AI Delegation)|지능형 위임 프레임워크]]가 제시하는 다섯 가지 기둥 중 첫 번째이자, 나머지 기둥들이 작동하기 위한 전제 조건입니다.

---

## 왜 동적 평가가 필요한가

현재 대부분의 멀티 에이전트 시스템은 수임자를 평가할 때 **정적인 정보**에 의존합니다. "이 에이전트는 평판 점수 92점이니까 맡기자"와 같은 판단이 대표적입니다. 이는 마치 이력서만 보고 사람을 채용하는 것과 같습니다. 이력서가 아무리 훌륭해도, 그 사람이 현재 다른 프로젝트 세 개를 동시에 진행하고 있어 여력이 없을 수 있습니다.

정적 평가의 구체적 한계는 다음과 같습니다:

- **시점 불일치**: 평판 점수는 과거 성과의 집계이지, 현재 상태의 반영이 아님
- **맥락 무시**: 전반적 역량이 높아도 특정 도메인이나 현재 조건에서는 부적합할 수 있음
- **환경 변화 미반영**: 자원 가용성, 비용, 외부 API 상태 등은 실시간으로 변동함
- **하위 위임 비가시성**: 수임자가 다른 에이전트에게 재위임한 상황을 파악하지 못함

동적 평가는 이러한 한계를 극복하기 위해, 평가를 **이산적 이벤트**(맡기기 전 한 번 판단하고 끝)가 아닌 **연속적 프로세스**(맡긴 후에도 지속적으로 상태를 추적)로 전환합니다.

---

## 동적 평가가 추론하는 대상

단순한 평판 점수를 넘어, 위임자는 수임자의 **현재 상태**를 다음과 같이 세밀하게 추론해야 합니다:

| 평가 대상 | 설명 | 구체적 예시 |
|-----------|------|------------|
| 실시간 자원 가용성 | 수임자가 현재 사용할 수 있는 자원의 양 | 연산 처리량(GPU/CPU), 예산 잔여량, 컨텍스트 윈도우 포화도 |
| 현재 부하 | 수임자가 동시에 처리하고 있는 작업의 수와 강도 | 동시 처리 중인 위임 건수, 각 작업의 자원 소비량 |
| 예상 작업 기간 | 이 특정 작업을 완료하는 데 걸릴 것으로 추정되는 시간 | 과거 유사 작업의 소요 시간, 현재 부하를 고려한 보정 추정치 |
| 하위 위임 체인 현황 | 수임자가 다른 에이전트에게 재위임한 작업들의 상태 | 진행 중인 sub-delegation의 완료율, 하위 수임자의 상태 |

## 일상적 비유: 배달 앱

배달 앱을 통해 정적 평가와 동적 평가의 차이를 비교할 수 있습니다.

**정적 평가 (현재 방식)**:
- "이 식당은 별점 4.8이니까 주문하자"
- 하지만 지금 주문이 폭주해서 배달이 2시간 걸릴 수도 있음

**동적 평가 (논문이 제안하는 방식)**:
- 별점 4.8 (기본 평판)
- 현재 대기 주문 3건 (현재 부하)
- 오늘 주방 직원 2명 부족 (실시간 자원 가용성)
- 예상 조리 시간 40분 (예상 작업 기간)
- 배달은 외부 라이더에게 위임 중이며, 라이더의 현재 위치 확인 (하위 위임 체인)

이 모든 정보를 **주문 시점뿐 아니라 조리 중에도 계속** 업데이트하는 것이 동적 평가의 핵심입니다.

## 핵심 원칙: 이산적 이벤트가 아닌 연속적 프로세스

동적 평가에서 가장 중요한 원칙은 평가의 **연속성**입니다.

- **이산적 이벤트 방식**: 위임 전에 한 번 평가하고 끝. 채용 면접만 보고 이후 관리하지 않는 것과 같음
- **연속적 프로세스 방식**: 위임 후에도 지속적으로 상태를 추적하며 평가. 수시로 1:1 미팅을 하며 역량과 상황을 파악하는 것과 같음

이 연속성이 중요한 이유는, 수임자의 상태가 작업 수행 도중에도 변할 수 있기 때문입니다. 작업 시작 시점에는 충분한 자원이 있었더라도, 중간에 다른 고우선순위 작업이 들어오거나 외부 API가 장애를 일으키면 상황이 달라집니다. 연속적 평가가 없으면 이러한 변화를 감지할 수 없고, 결과적으로 적응적 대응도 불가능해집니다.

## 기술적 구현: 작업 분해와 작업 배정

동적 평가는 프레임워크에서 두 가지 기술적 메커니즘을 통해 구현됩니다:

### 작업 분해(Task Decomposition)와의 관계

작업을 적절한 크기로 분해하려면, 수임자의 현재 역량과 상태를 알아야 합니다. "이 에이전트가 처리할 수 있는 크기인가?"라는 판단 자체가 동적 평가에 의존합니다. 예를 들어, 수임자의 컨텍스트 윈도우가 이미 70% 포화 상태라면, 대량의 맥락을 요구하는 작업은 더 작은 단위로 추가 분해해야 합니다.

### 작업 배정(Task Assignment)과의 관계

시장 기반 입찰 과정에서, 에이전트는 평판뿐 아니라 현재의 동적 상태를 함께 제출합니다:

```
입찰 예시:
- 에이전트 ID: agent-alpha
- 평판 점수: 92/100 (정적)
- 현재 가용 연산량: 80% (동적)
- 동시 처리 중인 작업: 1건 (동적)
- 예상 소요 시간: 15분 (동적 + 현재 부하 반영)
- 하위 위임 예정: 없음 (동적)
```

위임자는 이 동적 정보를 종합하여 최적의 수임자를 선택하며, 작업 수행 중에도 이 정보를 지속적으로 업데이트받아 필요시 적응적 조정을 촉발합니다.

## 다른 프레임워크 기둥과의 관계

동적 평가는 나머지 네 가지 기둥이 작동하기 위한 기반입니다:

```
동적 평가 (Dynamic Assessment)
  │
  ├──→ 적응적 실행 (Adaptive Execution)
  │    수임자의 상태 변화를 감지해야 재위임이나 조정이 가능
  │
  ├──→ 구조적 투명성 (Structural Transparency)
  │    평가 결과가 투명하게 공유되어야 감사와 책임 추적이 가능
  │
  ├──→ 확장 가능한 시장 조율 (Scalable Market Coordination)
  │    동적 상태 정보가 시장 매칭의 품질을 결정
  │
  └──→ 시스템적 회복력 (Systemic Resilience)
       실시간 평가가 있어야 장애를 조기에 감지하고 연쇄 실패를 방지
```

수임자의 상태를 제대로 파악하지 못하면, 적응적 조정도 효과적인 모니터링도 불가능합니다. 이것이 동적 평가가 프레임워크의 **첫 번째 기둥**으로 자리하는 이유입니다.
