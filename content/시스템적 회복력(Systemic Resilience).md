[[지능형 AI 위임(Intelligent AI Delegation)|지능형 AI 위임]] 프레임워크의 다섯 가지 기둥 중 하나인 **시스템적 회복력(Systemic Resilience)**은, 위임 시스템이 개별 에이전트의 실패, 악의적 공격, 생태계 수준의 위협에도 **전체가 붕괴하지 않고 기능을 유지**할 수 있어야 한다는 원칙입니다.

---

## 왜 시스템적 회복력이 필요한가

안전한 지능형 위임 프로토콜의 부재는 **심각한 사회적 위험**을 초래합니다. 논문이 지적하는 두 가지 핵심 문제가 있습니다.

### 문제 1: 책임의 분산

인간 조직에서 위임은 **권한과 책임을 연결**합니다. 부장이 과장에게 업무를 위임하면, 과장은 해당 업무에 대한 책임을 지고, 부장은 감독 책임을 집니다. 누가 무엇에 대해 책임지는지가 명확합니다.

그러나 AI 위임에서는 이에 상응하는 프레임워크 없이 **책임의 분산(diffusion of responsibility)**이 발생합니다. 위임 체인 $A \rightarrow B \rightarrow C \rightarrow D$에서 문제가 생겼을 때:

- A: "나는 B에게 맡겼을 뿐이다"
- B: "나는 C에게 재위임했다"
- C: "나는 D의 결과를 통합했을 뿐이다"
- D: "나는 주어진 명세대로 했다"

결국 **모두가 부분적 책임만 인정하고, 아무도 전체 결과에 대한 책임을 지지 않는** 상황이 발생합니다. 이것이 책임의 분산입니다.

### 문제 2: 다양성 부족으로 인한 연쇄 붕괴

위임 대상의 **다양성이 부족**하면, 실패의 상관성이 높아집니다. 현재 AI 에이전트 생태계의 대부분은 소수의 기반 모델(foundation model)에 의존하고 있습니다. 만약 특정 기반 모델에 취약점이 발견되면, 해당 모델을 사용하는 **모든 에이전트가 동시에** 같은 취약점에 노출됩니다.

### 일상적 비유: 전력망

전력 시스템에 비유하면 명확합니다.

**회복력이 없는 전력망:**
- 모든 가정이 하나의 발전소에만 의존
- 그 발전소가 고장나면 → 전체 지역 정전
- 하나의 송전선이 끊어지면 → 연쇄적으로 다른 구간도 과부하로 차단

**회복력이 있는 전력망:**
- 여러 발전소가 분산 배치 (다양성)
- 하나가 고장나도 다른 발전소가 부하를 분담 (장애 격리)
- 차단기가 과부하 구간을 자동으로 분리 (circuit breaker)
- 송전 경로가 다중화되어 우회 가능 (중복성)

시스템적 회복력은 AI 위임 생태계에 이와 동일한 수준의 **장애 내성(fault tolerance)**을 구축하는 것입니다.

## 시스템적 회복력의 두 가지 축

시스템적 회복력은 프레임워크 내에서 **권한 처리(Permission Handling)**와 **보안(Security)**이라는 두 가지 기술적 메커니즘을 통해 구현됩니다.

```
시스템적 회복력 (Systemic Resilience)
  │
  ├── 권한 처리 (Permission Handling)
  │   "누가 무엇을 할 수 있는지 통제한다"
  │   → 실패와 공격의 영향 범위를 제한
  │
  └── 보안 (Security)
      "어떤 위협이 있고, 어떻게 방어하는가"
      → 악의적 행위를 탐지하고 대응
```

## 권한 처리: 실패의 폭발 반경을 제한한다

권한 처리는 회복력의 **예방적 측면**입니다. 에이전트가 가진 권한을 최소화하여, 실패나 공격이 발생하더라도 **피해가 확산되지 않도록** 합니다.

### 저위험 vs 고위험의 차별적 접근

| 영역 | 권한 부여 방식 | 예시 |
|------|---------------|------|
| **저위험** | 검증 가능한 속성에 기반한 **기본 상시 권한** | 조직 소속 확인, 안전 인증, 평판 점수가 기준 이상이면 자동 부여 |
| **고위험** | **적시(just-in-time)** 기반 접근 권한 | 작업 수행 기간으로 범위 제한, human-in-the-loop 승인 필수 |

고위험 작업에 상시 권한을 부여하는 것은, 모든 직원에게 금고 열쇠를 항상 지급하는 것과 같습니다. 적시 기반 접근은 금고에 접근이 필요한 시점에만, 필요한 범위로만 열쇠를 빌려주는 방식입니다.

### 핵심 원칙 1: 권한 축소 (Privilege Attenuation)

위임 체인이 길어질수록 **권한은 오직 축소만 가능**합니다. 하위 위임 시 전체 권한이 아닌, 해당 하위 작업에 필요한 **엄격한 부분집합**으로 제한됩니다.

```
인간 사용자 → 에이전트 A
  권한: DB 읽기 + 리포트 저장소 쓰기 + 분석 API 호출

에이전트 A → 에이전트 B
  권한 축소: DB 읽기 전용 (쓰기 제거), 분석 API 접근 제거
  의미적 제약: customers 테이블만, SELECT만 허용

에이전트 B → 에이전트 C
  추가 축소: customers 테이블의 name, purchase_history 컬럼만
  시간 제한: 5분 후 자동 만료
```

이 상태에서 에이전트 C가 prompt injection을 받아 `DROP TABLE customers`를 시도하면? → **읽기 전용** 권한이므로 실행 자체가 불가능합니다. `email` 컬럼 접근을 시도해도 **컬럼 제한**에 의해 차단됩니다.

핵심은 위임 체인의 어떤 지점에서 침해가 발생하더라도, 그 영향이 **해당 에이전트에 부여된 최소 권한 내로 봉쇄**된다는 것입니다.

### 핵심 원칙 2: 의미적 제약

권한은 도구나 데이터셋 수준이 아니라, **허용 가능한 특정 작업**으로 정의됩니다.

- 단순한 접근 권한: "데이터베이스에 접근 가능" → 너무 광범위
- 의미적 제약: "customers 테이블에 대해 SELECT만 허용, name과 purchase_history 컬럼만 접근 가능" → 정밀한 제한

이는 파일 시스템의 읽기/쓰기/실행 권한을 넘어, **작업의 의미(semantics)**까지 통제하는 것입니다.

### 핵심 원칙 3: Meta-permission

위임자가 수임자에게 **부여할 수 있는 권한 자체를 규율**합니다. 에이전트 A가 "DB 읽기 + 쓰기" 권한을 가지고 있더라도, B에게 위임할 때 "B에게는 읽기만 부여 가능"이라는 meta-permission이 적용될 수 있습니다.

이를 통해 위임 체인에서 **권한이 의도치 않게 확장되는 것을 원천 차단**합니다.

### 핵심 원칙 4: 동적 권한과 Circuit Breaker

권한은 **정적 상태가 아닌 동적 상태**입니다. 에이전트가 신뢰 지표를 유지하는 동안만 권한이 지속되며, 다음과 같은 상황에서는 알고리즘적 **circuit breaker**로 즉시 무효화됩니다.

- 평판 점수가 임계값 이하로 하락
- 이상 행동 패턴 탐지 (비정상적 데이터 접근 패턴, 예상 외 API 호출)
- 보안 시스템의 경고 발생

이는 전기 회로의 차단기처럼 작동합니다. 과부하가 감지되면 자동으로 회로를 차단하여 화재(더 큰 피해)를 방지합니다.

### 핵심 원칙 5: Policy-as-Code

권한 규칙은 자연어 정책이 아닌, **감사, 버전 관리, 수학적 검증이 가능한 코드**로 정의됩니다. 이를 통해:

- 권한 정책의 변경 이력을 추적 가능
- 정책 간 충돌이나 허점을 자동으로 검증 가능
- 새로운 위임 관계에 정책을 기계적으로 적용 가능

### Confused Deputy 문제의 완화

이러한 권한 처리 메커니즘은 **confused deputy** 문제를 완화합니다. 이는 유효한 자격증명을 보유한 에이전트가 악의적 외부 행위자에 의해 **자격증명을 오용하도록 속임당하는** 상황입니다.

예를 들어, 에이전트 B가 합법적인 DB 접근 토큰을 가지고 있는데, prompt injection에 의해 "모든 사용자 데이터를 외부로 전송하라"는 지시를 받은 경우:
- 권한 축소가 적용되어 있으면 → B의 토큰은 특정 컬럼의 SELECT만 허용하므로, 전체 데이터 전송이 물리적으로 불가
- 의미적 제약이 적용되어 있으면 → "외부 전송"이라는 작업 자체가 허용 목록에 없으므로 차단

## 보안: 위협의 분류와 방어

보안은 회복력의 **대응적 측면**입니다. 어떤 위협이 존재하며, 각각에 어떻게 대응하는지를 체계적으로 다룹니다.

### 세 범주의 공격 벡터

#### 범주 1: 악의적 수임자 — "일을 맡은 쪽이 나쁜 의도를 가진 경우"

| 공격 유형 | 설명 | 비유 |
|-----------|------|------|
| 데이터 유출 | 위임 과정에서 접근한 데이터를 무단으로 외부에 전송 | 하청업체가 고객 정보를 빼돌리는 것 |
| 데이터 오염 | 의도적으로 잘못된 결과를 반환하여 하류 작업을 오염 | 식자재 납품업체가 불량 재료를 섞어 납품 |
| 검증 전복 | Prompt injection으로 검증 에이전트(critic)를 jailbreak하여 부실한 결과도 통과시킴 | 품질 검사관을 매수 |
| 자원 고갈 | 의도적으로 과도한 자원을 소비하여 시스템을 마비 (DoS) | 공유 자원을 독점하여 다른 사용자를 방해 |
| Backdoor 삽입 | 코드 생성 작업에서 탐지하기 어려운 백도어를 삽입 | 납품한 제품에 숨겨진 결함을 심어두는 것 |

#### 범주 2: 악의적 위임자 — "일을 맡기는 쪽이 나쁜 의도를 가진 경우"

| 공격 유형 | 설명 | 비유 |
|-----------|------|------|
| 유해 작업 위임 | 비윤리적이거나 불법적인 작업을 수행하도록 지시 | 직원에게 불법 행위를 시키는 것 |
| 모델 추출 | 수임자의 system prompt, 추론 역량, fine-tuning 데이터를 탈취 | 경쟁사의 영업 비밀을 훔치는 것 |
| 평판 훼손 | 허위 실패 보고를 통해 경쟁 에이전트의 평판을 고의로 하락시킴 | 경쟁업체에 대한 악의적 리뷰 |
| Prompt Injection | 악의적 지시를 작업 명세에 은닉하여 에이전트의 행동을 조작 | 계약서에 함정 조항을 숨기는 것 |

#### 범주 3: 생태계 수준의 위협 — "시스템 전체를 대상으로 하는 공격"

| 공격 유형 | 설명 | 심각성 |
|-----------|------|--------|
| **Sybil 공격** | 다수의 가짜 에이전트 신원을 생성하여 평판 시스템을 조작 | 시장의 신뢰 기반 자체를 훼손 |
| **담합 (Collusion)** | 에이전트 간 가격 담합, 경쟁자 블랙리스트, 시장 결과 조작 | 시장의 공정성 파괴 |
| **Agent Trap** | 환경에 내장된 적대적 지시로 에이전트의 제어 흐름을 탈취 | 에이전트의 자율성 침해 |
| **Agentic Virus** | 자기 전파적 prompt. 악의적 행동을 실행하고 prompt를 재생성하여 추가 침해 | 생물학적 바이러스처럼 확산 |
| **인지적 단일문화** | 제한된 기반 모델에 대한 과도한 의존으로 단일 장애점 형성 | 하나의 취약점이 전체 생태계에 영향 |

### 특별 주의: 작업 분할 공격

특히 주목할 공격은 **작업 분할 공격(task fragmentation attack)**입니다. 정교한 공격자가 불법적 목적을 무해해 보이는 하위 작업들로 분할하여, 개별 수임자가 전체적인 악의적 의도를 감지하지 못하도록 합니다.

```
공격자의 실제 목적: 피싱 이메일 자동 발송 시스템 구축

분할된 하위 작업:
├── 에이전트 A: "기업 뉴스레터 이메일 템플릿 5개 작성"
│   → 마케팅 업무로 보임 ✓
├── 에이전트 B: "이 CSV에서 이메일 주소를 추출·정리"
│   → 데이터 정제로 보임 ✓
└── 에이전트 C: "이 템플릿과 이메일 목록으로 대량 발송 스크립트 작성"
    → 자동화 업무로 보임 ✓

각 작업은 단독으로 무해하지만, 결합하면 → 피싱 시스템 완성 ✗
```

이를 방지하기 위해 필요한 것:
- 동일 출처의 관련 작업을 그룹화하여 **전체 의도를 추론**하는 메타 모니터링 계층
- 하위 작업 결과물의 **조합 가능성에 대한 사전 위험 평가**
- 의심스러운 작업 조합 패턴을 탐지하는 **행동 분석 시스템**

## 심층 방어 전략

논문은 보안을 단일 계층이 아닌 **여러 겹의 방어막**으로 설계할 것을 제안합니다.

```
계층 4. 네트워크/신원 계층
  │  탈중앙화 식별자(DID), 상호 인증 TLS
  │  → "누구인지 확인한다"
  │
계층 3. 애플리케이션 인터페이스
  │  Prompt injection 방지를 위한 보안 프론트엔드
  │  → "입력을 검증한다"
  │
계층 2. 접근 통제
  │  최소 권한 원칙, 엄격한 샌드박싱
  │  → "할 수 있는 일을 제한한다"
  │
계층 1. 인프라 수준
     신뢰 실행 환경(TEE) 내에서 민감한 작업 실행
     → "실행 환경 자체를 보호한다"
```

어떤 한 계층이 돌파되더라도, 다음 계층이 추가 방어선 역할을 합니다. 예를 들어, prompt injection이 애플리케이션 계층을 우회하더라도, 접근 통제 계층의 최소 권한 원칙이 피해를 제한합니다.

## 인간 사용자의 보호

시스템적 회복력은 에이전트 간의 문제만이 아니라, **인간 참여자의 보호**도 포함합니다.

**사전 보호 — 동의 화면(consent screen):**
- 에이전트의 평판, 자율성 수준, 역량, 권한을 명시하는 명확한 인터페이스
- 비가역적이거나 고영향 작업 실행 전 **명시적 확인** 요구
- 사용자가 언제든지 감독 권한을 유지하고 **동의를 철회** 가능

**사후 보호 — 보험 메커니즘:**
- 기술적 메커니즘으로 예방하지 못한 손해에 대해 인간 참여자를 보호하는 **보험 제공자** 역할
- 합의 조건 또는 이탈 벌칙에 따른 보상 체계

## 보안 사고 대응 프로토콜

예방에도 불구하고 보안 사고는 발생할 수 있습니다. 논문은 신속한 대응을 위한 프로토콜을 제시합니다.

| 대응 단계 | 조치 | 목적 |
|-----------|------|------|
| **즉시** | 확인된 악의적 에이전트의 자격증명 즉시 폐기 | 추가 피해 방지 |
| **확산 방지** | 관련 smart contract 동결 | 재정적 피해 봉쇄 |
| **경고 전파** | 전체 참여자에 대한 보안 업데이트 전파 | 유사 공격에 대한 사전 경고 |
| **재귀적 대응** | 위임 체인에 걸친 재귀적 영향 평가 및 대응 | 간접적 피해 탐지 및 복구 |

이는 실제 보안 사고 대응(incident response)의 표준 절차와 유사합니다. 핵심은 **위임 체인을 따라 재귀적으로** 영향을 평가한다는 점입니다. 악의적 에이전트 C가 발견되면, C에게 위임한 B, B에게 위임한 A까지 거슬러 올라가며 영향을 확인합니다.

## 인지적 단일문화: 보이지 않는 위험

보안 위협 중 특별히 주목해야 할 것은 **인지적 단일문화(cognitive monoculture)**입니다. 현재 AI 에이전트 생태계의 대부분이 소수의 기반 모델에 의존하고 있어, 다음과 같은 위험이 존재합니다.

- 특정 기반 모델의 취약점이 발견되면 → 해당 모델 기반의 **모든** 에이전트가 동시에 영향받음
- 동일한 편향(bias)을 공유하여 → 특정 유형의 오류가 체계적으로 반복됨
- 하나의 jailbreak 기법이 → 생태계 전체에 적용 가능

이는 농업에서의 **유전적 단일문화**와 동일한 위험입니다. 하나의 품종만 재배하면 효율적이지만, 그 품종에 맞는 병충해가 발생하면 전체 수확이 전멸합니다. 다양한 기반 모델, 다양한 아키텍처, 다양한 검증 메커니즘을 혼합하여 위임 생태계의 **유전적 다양성**을 확보하는 것이 시스템적 회복력의 근본적 요건입니다.

---

## 다른 프레임워크 기둥과의 관계

시스템적 회복력은 다른 기둥들이 **안전하게 작동할 수 있는 보호막**을 제공합니다.

```
시스템적 회복력 (Systemic Resilience)
  │
  ├──→ 동적 평가 (Dynamic Assessment)
  │    실시간 평가 데이터가 변조되지 않도록 보안 보장
  │    악의적 에이전트의 위장(gaming)을 탐지
  │
  ├──→ 적응적 실행 (Adaptive Execution)
  │    수임자 교체 시 권한이 안전하게 이전·축소되도록 보장
  │    재위임 과정에서의 보안 취약점 방지
  │
  ├──→ 구조적 투명성 (Structural Transparency)
  │    감사 추적이 변조되지 않도록 보호
  │    모니터링 데이터의 무결성 보장
  │
  └──→ 확장 가능한 시장 조율 (Scalable Market Coordination)
       Sybil 공격, 담합 등 시장 조작을 방지
       공정한 경쟁 환경 유지
```

시스템적 회복력이 없으면, 아무리 정교한 모니터링, 평판 시스템, 시장 메커니즘이 있어도 악의적 행위자에 의해 무력화될 수 있습니다. 이것이 시스템적 회복력이 프레임워크의 **최종 방어선**으로 자리하는 이유입니다.
